# Apache APISIX

目标仓库：https://github.com/apache/apisix

## 1. 任务要求、问题来源和建议方向

你要在 Apache APISIX 中寻找一个真实的网关或插件贡献点。APISIX 是高性能云原生 API 网关和 AI Gateway，用插件处理认证、限流、观测、流量治理、请求改写、日志、安全策略和 LLM API 管理。作为 Apache 项目，它要求公开、可测试、可维护的改动。

你可以优先参考这些问题来源：

- 已有 issue、mailing list、PR review 中确认的插件 bug、schema 缺陷、文档错误或测试缺口。
- 某个插件的配置项、错误提示、优先级、执行 phase 或边界行为与文档不一致。
- AI Gateway、认证、日志、观测等方向已有相邻能力，但缺少清晰的小功能或示例。
- test-nginx 用例、集成测试或配置验证暴露出的回归风险。

建议修改方向：

- 优先给已有插件补 schema 校验、边界测试、文档示例、错误处理或兼容性修复。
- 新插件最好先有 issue 或社区讨论，说明为什么适合进入 APISIX，而不是只作为自定义插件存在。
- 对 agent 系统，可以考虑 AI Gateway 相关的小功能，例如 token 统计、审计字段、模型路由或请求安全检查。
- 网关处于生产流量入口，任何外部调用、超时、重试、日志和性能影响都要写清楚。

## 2. 交付要求和最终目标

你需要交付：

- 问题来源和证据说明，包括 issue、配置复现、测试失败、文档差异或社区讨论链接。
- 方案设计和影响范围，说明插件、schema、phase、priority、性能和兼容性影响。
- 代码或文档改动，包括 Lua 插件、测试用例、文档、示例配置或 changelog。
- 测试和验证结果，记录 test-nginx、单插件测试、lint 或手动请求结果。
- PR 标题和 PR 描述，符合 Apache 社区风格，说明问题、方案、验证和风险。

最终目标：提交一个范围小、证据足、测试完整的 APISIX PR，并通过维护者 review。
