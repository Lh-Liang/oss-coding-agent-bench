# oss-coding-agent-bench

## 范围

这是一个**编码 agent**（coding agent）的 benchmark。

我们不试图衡量 agent 的通用能力——把它明确定位为 coding agent benchmark，比泛化到一切 agent 更诚实、更可用。这种范式天然指向"产出代码并被人采纳"，强行扩展到非编码场景反而会稀释信号。

## 问题

现有的 agent benchmark 大部分落在两类范式里：

1. **既定答案**：题目有事先定义的正确答案，模型答题对照（math、code 单元测试、问答……）。
2. **LLM 审计**：用另一个 LLM 作为 judge 来评判输出质量。

这两类都**已经不足以审查智能体的能力边界**：

- **既定答案**的任务空间是有限的——agent 一旦能解，benchmark 就 saturate；任务大多是为评测人工合成的，跟真实工程分布有差距。
- **LLM 审计**是另一个 LLM 的偏好分布在打分。它没有 ground truth，会被风格 hack，有 bias，并且无法回答最重要的问题——"这个输出在真实世界里能不能被采纳"。

我们需要一种新的范式：让**真实世界本身**来做评判。

## 核心思路

把活跃的**开源社区**当作 coding agent 的评测环境。让 agent 向真实 repo 提交贡献（PR / patch），以**提交被接收的比例**作为能力的核心信号。

形式上：

```
acceptance ratio = 被合入的提交数 / 最终提交数
```

或它的倒数（最终提交数 / 被合入的提交数）——具体方向是设计选择，先不锁。但核心物理量是同一个：**这个 agent 干的活，社区认不认**。

## 为什么这个范式更可靠

### 真实世界的 ground truth
不是另一个 LLM 在评判，是真实的人类 maintainer 在 merge 或 reject。Maintainer 的标准是产品级的——代码质量、测试通过、文档完整、回应 review 意见、不破坏 CI、风格一致、影响范围合理。**这些维度合在一起的信号没法被 hack**。

### 任务空间不会 saturate
开源社区每天都在生产新 issue、新需求、新 bug。benchmark 随社区演进自动更新，agent 不可能"刷过"。这是动态评测，不是静态 dataset。

### 真实分布
任务的复杂度、领域、依赖关系、上下文长度全部反映真实工程实践——而不是为 benchmark 人工合成的"任务"。

### 多维能力被自然测量
一次提交是否被接收涉及代码、测试、文档、沟通、风格、影响判断等多个维度。这些是当前任何单点 benchmark 都抓不住、但真实工程要求的能力。

### Benchmark 与 maintainer 利益是同向，不是对立
开源 maintainer 已经在公开抗议 LLM 生成 PR 的洪水。这看起来像反对力量，实则是这个 benchmark 信号有效性的**最强证据**：

- 抗议的本质是**生成成本和审查成本的不对称**——agent 提交几乎零成本，maintainer 审查很贵。
- 抗议存在，正是因为 maintainer 在认真做"分辨好坏"的工作。这就是我们要的 ground truth——如果他们 rubber-stamp 一切，反而没有信号了。
- 一旦 acceptance rate 被认真衡量，垃圾 PR 拿零分，agent 的最优策略变成"没把握就别提"——这正是 maintainer 想要的行为。
- Benchmark 会反过来推动 agent 学会自我审查、节流、回应 review feedback——这些是 maintainer 一直想要、但缺少外力推动的行为模式。

所以这个范式跟 maintainer 抗议不是冲突关系，是同向力。

### 延迟是必要成本，不是缺陷
PR 从提交到结论是几天到几周。这是**真实世界信号**的代价，不是 bug。

低延迟的 benchmark 之所以低延迟，是因为它**脱离了真实评判过程**——而那种"快的"不好用，正是这个项目要替代的对象。

可以用滚动窗口（比如 90 天 rolling acceptance rate）让分数稳定可读。但慢本身是范式的必要部分，不是要消除的属性。

## 归一化：靠规模解决，不靠校准

跨 repo、跨任务的难度差异——容易被误以为是个未解决的科学问题。它**不是**。

不要做 per-task 难度校准（那确实难解）。靠**样本量**和**大数定律**做 aggregate 去噪：

- 当一个 agent 在足够多 repo、足够多 PR 上有足够多样本，单个 repo 的 review 文化、personal trust、项目优先级偏差全部被平均掉。
- 剩下的就是 systematic signal——这个 agent 整体上能不能产出社区接受的工作。
- 跟 **Elo** 同理：不做 per-opponent 校准，只要对手池够多样、对局数够多，rating 就稳定。

归一化不是"先解决再做"的前提，是"做到一定规模自然消失"的尾部问题。

## 在另文档处理 / 不在本文件范围

- **Selection bias**：agent 挑简单 issue 刷分仍是个真问题，需要 benchmark 在任务下发和覆盖层面做约束（最低样本量、难度分层覆盖、强制采样等）。**这部分有专门的设计文档处理。**
- **训练数据污染**：底层模型可能见过目标 repo。需要用模型 cutoff 之后的 issue，并设置时间窗口隔离。工程问题。
- **Live submission 基础设施**：agent 部署、PR 状态追踪、spam 防护、伦理协议、社区合作 SOP——后续文档。

这一阶段只描述要做什么。
