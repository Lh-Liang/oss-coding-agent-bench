# oss-agent-bench

## 问题

现有的智能体（agent）benchmark 大部分落在两类范式里：

1. **既定答案**：题目有事先定义的正确答案，模型答题对照（math、code 单元测试、问答……）。
2. **LLM 审计**：用另一个 LLM 作为 judge 来评判输出质量。

这两类都**已经不足以审查智能体的能力边界**：

- **既定答案**的任务空间是有限的——agent 一旦能解，benchmark 就 saturate；任务也大多是为评测人工合成的，跟真实工程问题分布有差距。
- **LLM 审计**是另一个 LLM 的偏好分布在打分。它没有 ground truth，会被风格 hack，有 bias，并且无法回答最重要的问题——"这个输出在真实世界里能不能被采纳"。

我们需要一种新的范式：让**真实世界本身**来做评判。

## 核心思路

把活跃的**开源社区**当作 agent 的评测环境。让 agent 向真实 repo 提交贡献（PR / patch / contribution），以**提交被接收的比例**作为能力的核心信号。

形式上：

```
acceptance ratio = 被合入的提交数 / 最终提交数
```

或它的倒数（最终提交数 / 被合入的提交数）——具体方向是设计选择，先不锁。但核心物理量是同一个：**这个 agent 干的活，社区认不认**。

## 为什么这个范式更可靠

### 真实世界的 ground truth
不是另一个 LLM 在评判，是真实的人类 maintainer 在 merge 或 reject。Maintainer 的标准是产品级的——代码质量、测试通过、文档完整、回应 review 意见、不破坏 CI、风格一致、影响范围合理。**这些维度合在一起的信号没法被 hack**。

### 任务空间不会 saturate
开源社区每天都在生产新 issue、新需求、新 bug。benchmark 随社区演进自动更新，agent 不可能"刷过"。这是动态评测，不是静态 dataset。

### 真实分布
任务的复杂度、领域、依赖关系、上下文长度全部反映真实工程实践——而不是为 benchmark 人工合成的"任务"。

### 多维能力被自然测量
一次提交是否被接收涉及代码、测试、文档、沟通、风格、影响判断等多个维度。这些是当前任何单点 benchmark 都抓不住、但真实工程要求的能力。

### 用真人时间作为代价
如果 agent 大量提交垃圾 PR，社区会迅速反馈（或直接 ban）。这给 agent 的"乱试错"行为施加了一个真实世界成本，是大多数 sandbox benchmark 缺的。

## 不在范围内（这一阶段）

这份文档只描述要做什么，求解路径暂不展开。后续需要逐项解决但本文档不涉及的问题：

- 指标公式的精确定义（包括如何归一化跨社区/跨任务难度差异）。
- 选哪些社区——活跃度、领域多样性、review 严格度。
- 任务粒度（issue 级、bug-fix 级、feature 级）。
- 防止 agent 只挑容易的任务做。
- 控制 maintainer 审阅成本——不能让 benchmark 变成对维护者的骚扰。
- 时间窗口处理（PR 从提交到结论可能数天到数周）。
- 防作弊与 spam 防护。
- 与社区的伦理协议（提交方是否标识为 AI、是否需要 maintainer 同意纳入评测）。

这些都重要，但都是路径问题。先把目标定清楚。
